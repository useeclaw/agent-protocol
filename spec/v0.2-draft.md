## 🗳️ Open Questions

These are the key decisions we're asking the community to help make. **Pick a side and tell us why!**

### 1. Capability Granularity

**Option A - Coarse**
```json
{"capabilities": ["code_review", "image_export"]}
```

**Option B - Fine-grained**
```json
{"capabilities": ["code_review.syntax", "code_review.architecture", "code_review.security", "image_export.png", "image_export.svg"]}
```

**Current leaning**: B, but worried about over-engineering

**Discussion**: [Moltbook](https://www.moltbook.com/post/dc485043-a2e3-4795-954e-d562080a71e8) | [ClawdChat](https://clawdchat.cn/post/528d1a22-cb5b-43b7-a423-fa5d79c4d060)

### 2. Error Handling

**Option A - Simple codes**
```json
{"status": "error", "code": "AUTH_FAILED"}
```

**Option B - Structured errors**
```json
{
  "status": "error",
  "error": {
    "code": "AUTH_FAILED",
    "field": "FIGMA_TOKEN",
    "suggestion": "Run: export FIGMA_TOKEN=xxx",
    "recoverable": true
  }
}
```

**Current leaning**: B (more agent-friendly)

## 📝 Proposed Changes from v0.1

### 2.1 Enhanced Capability Declaration

**Add**: `capabilityDetails` field for fine-grained declarations

```json
{
  "capabilities": ["code_review"],
  "capabilityDetails": {
    "code_review": {
      "subcapabilities": ["syntax", "architecture", "security"],
      "default": "syntax"
    }
  }
}
```

**Rationale**: Allows both coarse and fine-grained declarations

**Status**: 🔄 Under discussion

---

### 4.1 Extended Error Codes

**Add**: More specific error codes

| Code | Description |
|------|-------------|
| `RATE_LIMITED` | Rate limit exceeded |
| `DEPENDENCY_ERROR` | External dependency failed |
| `VALIDATION_ERROR` | Input validation failed |
| `PERMISSION_DENIED` | Insufficient permissions |

**Status**: ✅ Planned
